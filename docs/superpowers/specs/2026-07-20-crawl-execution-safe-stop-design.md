# 抓取执行权与安全停止设计

> 状态：已审核并实现。

## 1. 目标

解决以下同一类并发问题：

- 点击停止后 OpenCLI 子进程被结束，但 worker 主流程仍可能继续；
- 停止后启动新抓取，旧执行仍然写日志、进度或业务数据；
- 重复或陈旧 Celery 消息会重跑已经完成、失败或停止的任务；
- 停止发生在搜索、OCR、模型提取、归档或重试等待期间时，响应不及时或最终被写成 `FAILED`。

## 2. 已确认的根因

### 2.1 worker 没有原子领取任务

当前 `run_crawl` 只排除不存在和 `STOPPED` 的任务，然后直接写成 `RUNNING`。因此 `COMPLETED`、`FAILED`、`STOP_REQUESTED` 等历史任务收到陈旧消息时仍会再次执行；两个相同消息也可能同时执行同一任务。

### 2.2 同一任务 ID 被重启复用

重启接口把原任务重新置为 `PENDING`，但旧执行和新执行共享同一 task ID。旧执行无法判断自己已经失去执行权，会继续提交数据库事务。

### 2.3 停止检查点过少

当前主要只在搜索全部完成后、每篇笔记开始前和所有笔记完成后检查停止。城市/关键词/博主搜索循环、笔记详情、下载、逐图 OCR、MiniMax 提取、归档和阶段重试等待中没有统一检查点。

### 2.4 PID 注册不能代表任务执行权

注册表只以 task ID 保存一个 PID；新旧执行会互相覆盖。停止函数未校验进程身份，最终也会无条件注销并返回成功。结束当前 OpenCLI 子进程不能自动阻止 Celery 函数继续运行。

### 2.5 异常路径会覆盖停止状态

停止杀死子进程后产生的普通异常可能进入兜底异常分支，将已经请求停止的任务改成 `FAILED`。

## 3. 状态与执行权设计

### 3.1 唯一执行令牌

每次首次执行或人工重启都生成新的 `run_token`。Celery 消息必须同时携带 `task_id` 和 `run_token`；所有进度、日志、计数和最终状态写入都验证数据库当前令牌仍与本执行一致。

旧执行一旦发现令牌不匹配，立即退出且不得再写任何内容。

### 3.2 原子领取

worker 只能通过单条条件更新领取任务：

`PENDING + run_token 匹配 -> RUNNING`

只有受影响行数为 1 的执行可以继续。重复消息、旧令牌消息，以及 `RUNNING/STOP_REQUESTED/STOPPED/FAILED/COMPLETED` 状态消息全部直接忽略并留下可审计日志；不得把历史任务重新置为 `RUNNING`。

### 3.3 安全停止语义

1. API 将当前执行写为 `STOP_REQUESTED`；
2. 若当前正阻塞在 OpenCLI 子进程，按 `task_id + run_token` 精确结束该子进程；
3. worker 在最近检查点结束当前原子步骤，不再领取下一个步骤；
4. worker 确认停止后写为 `STOPPED`；
5. 停止路径不计作单篇失败或任务失败。

一期维持单 worker、`solo`、并发 1 的部署方式。新的抓取可以先创建为 `PENDING`，但必须等待旧执行确认 `STOPPED` 后才能领取；界面展示“正在安全停止上一任务”。

### 3.4 检查点

统一的 `assert_execution_active` 检查至少放在：

- 每个城市、关键词和博主搜索前后；
- 每篇笔记详情获取前后；
- 下载前后；
- 每张图片 OCR 前后；
- MiniMax 调用前后；
- 每次阶段重试前和重试等待期间；
- 每个子活动入库前；
- 归档生成前后；
- 最终状态写入前。

无法中断的单次库调用返回后，必须先检查执行权再提交结果。

### 3.5 事务边界

单篇推文作为最小业务事务。执行权失效或停止时：

- 已完整完成的历史推文保留；
- 当前未完成推文回滚或保持明确的可恢复状态，不伪装成 `PROCESSED`；
- 不再增加下载、OCR、提取、失败或跳过计数；
- 不再生成归档、重复候选或后续日志。

### 3.6 子进程注册表

注册键改为 `task_id + run_token`，记录 PID、启动时间和可验证的命令身份。结束前校验 PID 仍属于当前执行；结束后确认进程实际不存在再返回成功。旧执行不得覆盖新执行的注册记录。

## 4. API 与前端行为

- “停止抓取”进入 `STOP_REQUESTED` 后按钮禁用并显示安全停止中；
- 确认 `STOPPED` 后允许“继续抓取”；
- “开始抓取”遇到活动任务时先请求停止，创建新任务但不与旧任务并行执行；
- 失败、等待登录和已停止任务的继续抓取都会生成新 `run_token`；
- 仪表盘展示当前阶段、当前推文、已发现/下载/OCR/提取/失败/跳过数量；
- 对重复消息、执行权丢失和安全停止分别记录清晰日志。

## 5. TDD 范围

### 后端

- 同一消息并发领取只有一个成功；
- 已完成、失败、停止和停止请求中的任务收到消息不会重跑；
- 旧令牌不能写进度、日志、计数、活动或最终状态；
- 分别在搜索、下载、OCR、MiniMax、重试等待、归档阶段请求停止；
- 杀死 OpenCLI 后任务最终为 `STOPPED` 而不是 `FAILED`；
- 停止旧任务后启动新任务，旧任务不再产生任何数据库或文件写入；
- PID 注册与注销必须匹配 task ID 和 run token。

### 前端组件与浏览器 E2E

- 停止、停止中、已停止和继续抓取按钮状态正确；
- 创建新任务时能看到上一任务安全停止过程；
- 页面轮询显示阶段细分并最终稳定；
- 用真实本地 worker 执行“启动 -> 阶段中停止 -> 再启动”，确认旧任务不再变化。

## 6. 验收标准

- 点击停止后，在没有不可中断调用的情况下 5 秒内进入 `STOPPED`；
- 发生不可中断调用时，调用返回后立即停止，期间不会提交调用结果；
- 停止后再次抓取，旧执行的任务日志、计数、Note、Activity 和归档文件均保持不变；
- 重复或陈旧 Celery 消息不会执行历史任务；
- 单篇推文处理失败不影响其他推文；
- 后端、前端组件和真实 worker E2E 全部通过。

## 7. 不在本次范围

- 不增加多 worker 并发抓取；
- 不迁移 Redis；
- 不使用强制结束整个 Celery worker 作为正常停止方案。
