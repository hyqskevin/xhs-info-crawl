# 推文维度活动管理、去重与周报设计

> 状态：已审核并实现。

## 1. 目标

以小红书推文 `Note` 作为活动信息的聚合根：活动管理首页一行代表一篇推文，推文图片识别出的多个活动作为详情子项。审核、去重、批量删除和周报导出都以推文为主要维度，同时保留对子活动的纠错能力。

## 2. 已确认的产品规则

- 一篇推文可以包含零到多个活动；
- 活动管理首页展示推文，不把一篇推文拆成多行活动；
- 详情在推文信息表格后展示来源页面图片、OCR 文本和全部子活动；
- 子活动允许编辑或删除错误识别结果；
- 保留“待审核”，但它是推文审核状态，不与下载/OCR/提取处理状态混用；
- 去重审核选择“保留 A/B”后，未保留的整篇推文及其子活动退出业务列表；
- 周报指“本周发布的推文周报”，按推文发布时间筛选，不按子活动发生时间筛选；
- 单次周报只支持一个城市；
- 入选推文下所有未删除子活动都导出，即使活动发生在其他周；
- Markdown 和 Excel 都要包含原文链接和图片链接。

## 3. 当前实现与问题结论

### 3.1 当前可以工作的部分

`Note -> Activity` 的一对多结构已经存在，且 `platform_note_id` 有数据库唯一约束。现有 7 条去重相关定向测试通过，证明完全相同 URL 的搜索结果能在同批结果内合并，活动相似度计算和候选创建的基本路径可以执行。

### 3.2 同一推文再次抓取不能被稳定、友好地过滤

当前搜索结果只按原始 URL 字符串去重，入库前也只按 `source_url` 精确查询。小红书同一篇推文在 `xsec_token`、查询参数或 URL 形式变化后不会命中该判断。

虽然数据库的 `platform_note_id` 唯一约束最终可能阻止重复入库，但它发生在详情获取之后，而且会以单篇处理异常结束。这属于“靠数据库报错兜底”，不是正常过滤：浪费详情下载/OCR前的资源，也会增加失败数量和错误日志。

另外，现有 `platform_note_id` 仅通过 URL 最后一个路径片段截取，不足以覆盖小红书的多种链接形式。

### 3.3 不完整推文判断过于宽松

已有推文只要存在任意一条 Activity，就会被强制标为 `PROCESSED` 并跳过重试。若上次只提取出部分活动，会被误判为完整，无法恢复。

### 3.4 当前活动级去重层级不合理

现有候选会把同城市的所有活动两两比较，没有排除相同 `note_id`。2026-07-20 对本地 `data/app.db` 的只读审计结果为：`pending=298`、`ignored=20`、`merged=16`，其中 303 个候选的 A/B 活动属于同一篇推文；这会把“合集中的多个真实活动”误当成重复。

候选合并后仅修改当前候选和未保留 Activity，没有统一关闭所有引用已删除/已合并活动的其他待处理候选，因此存在悬空待审核记录；同次审计确认 110 个 pending 候选指向不存在、已删除或已合并的 Activity。`matched_fields` 模型是 JSON 列表，但创建逻辑写入逗号字符串，也需要修正。

## 4. 数据模型

### 4.1 Note 聚合根

在 `Note` 中明确分离两类状态：

- `processing_status`：下载、OCR、提取、完成、部分失败；
- `review_status`：`PENDING`、`APPROVED`、`REJECTED`、`MERGED`、`DELETED`。

保留 `published_at` 和 `created_at`。必要时增加 `updated_at`、`deleted_at`、`merged_into_note_id`，用于审计和软删除。

历史数据迁移采取保守规则：只有该推文下全部有效子活动均为已审核/已发布时，推文才迁移为 `APPROVED`；只要存在 `RAW`、`NEEDS_REVIEW` 或无法判断的子项，推文保持 `PENDING`。

### 4.2 子活动

Activity 继续关联 `note_id`，保存名称、时间、地点、费用、类型、摘要和来源图片索引。子活动状态只表达本条识别结果是否有效，不再承担整篇推文审核状态。

### 4.3 推文重复候选

新增独立的 `NoteDuplicateCandidate`，保存 A/B 推文、相似度、匹配字段、状态、处理结果和保留推文 ID。不再为新数据创建 Activity 级重复候选。

历史 Activity 候选在迁移时归档或关闭；涉及相同 `note_id`、已删除/已合并 Activity 或不存在记录的 pending 候选全部失效，不进入新审核列表。

## 5. 精确去重设计

### 5.1 统一身份解析

实现唯一的 XHS 推文身份解析器，输入搜索 URL、分享 URL 或详情 URL，输出：

- `platform_note_id`：稳定主身份；
- `canonical_url`：移除 query、fragment 和追踪参数后的标准地址；
- 原始可访问 URL：保留当前有效 `xsec_token`，供 OpenCLI 打开详情。

解析器必须覆盖项目实际出现的 `/explore/{id}`、`/discovery/item/{id}` 等受支持形式；解析失败时不猜测 ID，记录结构化警告并按原 URL 谨慎处理。

### 5.2 三道过滤

1. 搜索结果聚合时，以 `city_code + platform_note_id` 合并不同关键词或博主命中的同一篇推文，并合并命中来源；
2. 获取详情和下载前，以 `platform_note_id` 查询数据库；已完整处理的推文记录为“历史推文已过滤”，不计失败；
3. 数据库唯一约束继续作为并发情况下的最后防线；若并发插入冲突，应转为重复跳过而不是单篇失败。

同一推文的新 URL 带有更新的有效 token 时，可以只更新访问 URL 和抓取元数据，不重复创建 Note、Activity、图片、OCR 或归档。

### 5.3 完整性判断

不得再以“存在任意 Activity”代表推文完成。完整性至少由处理状态、图片/OCR阶段结果、提取完成标记和归档结果共同决定。

历史完整推文直接跳过；明确不完整的推文进入幂等恢复流程，清理或覆盖当前未完成阶段的数据，不能重复追加部分结果。

## 6. 模糊去重设计

精确身份不同但内容疑似重复时，按推文级生成候选：

- 先以城市作为候选范围；
- 比较标题、正文、OCR 文本和子活动摘要；
- 子活动用于增强判断，但同一篇推文内部的活动永不互相比较；
- 采用可解释的匹配字段和阈值，低于阈值不进入人工审核；
- 候选对使用有序唯一键，避免重复创建。

点击“保留 A”或“保留 B”后：

- 保留推文及其有效子活动；
- 未保留推文设为 `MERGED` 并指向保留推文，其子活动通过父级自动退出列表和导出；
- 所有涉及未保留推文的其他 pending 候选同步关闭或重算；
- 不物理删除原始数据，保留审计与误操作恢复能力。

“忽略”只表示当前 A/B 不重复，两篇推文均保留。

## 7. 活动管理界面与 API

### 7.1 列表

一行一篇推文，包含：选择框、标题、城市、发布时间、识别活动数、审核状态和操作。支持城市、发布时间范围、审核状态筛选、分页和推文级批量审核/删除。

### 7.2 详情

宽抽屉或对话框依次展示：

1. 推文基本信息、正文和原文链接；
2. 来源页面图片；
3. 每张图片的 OCR 文本和识别状态；
4. 子活动表格，支持编辑和删除错误项。

### 7.3 API 边界

新增或调整为推文主资源：

- 推文分页列表与详情；
- 推文批量审核、删除；
- 子活动编辑、删除；
- 推文重复候选列表、保留 A/B、忽略；
- 图片通过受控静态资源接口返回，不暴露任意本地路径。

不提供手工新增推文或新增活动入口。

## 8. 本周推文周报

### 8.1 筛选

- 城市单选；
- 周选择器确定自然周；
- 以 `Note.published_at` 落在该周为准；
- 历史数据缺少发布时间时暂用 `created_at`，并在报告标记“发布时间待确认”；
- 只包含 `review_status=APPROVED` 且未删除、未合并的推文。

### 8.2 Markdown

一篇推文一个章节，包含标题、发布时间、城市、正文、OCR、原文链接、图片链接，以及该推文下全部未删除子活动。

### 8.3 Excel

一篇推文一行，至少包含标题、发布时间、城市、原文 URL、正文、OCR、图片链接、子活动数量、子活动详情。多个子活动放在同一单元格中分行展示，保持推文维度。

周报列表同时显示推文数和子活动数；预览与 Markdown/Excel 下载必须使用同一份查询结果。

## 9. TDD 范围

### 后端

- 同一 `platform_note_id` 的不同 `xsec_token`、查询参数和 URL 形式只处理一次；
- 历史推文命中发生在详情、下载、OCR 和模型调用之前，且不增加失败数；
- 唯一约束并发冲突转为幂等跳过；
- 不完整推文能够恢复，不能因存在一个子活动就误判完成；
- 同一推文内多个活动不生成重复候选；
- 推文候选不重复，保留/忽略后关联候选状态一致；
- 历史悬空候选被关闭；
- 周报按推文发布时间、单城市和审核状态筛选，并包含全部有效子活动；
- Markdown 与 Excel 的推文集合、数量、链接和子活动一致。

### 前端组件

- 活动管理一行一篇推文并正确显示活动数；
- 城市、时间、状态、分页和批量操作正确；
- 详情宽度、图片、OCR 和子活动表格正确；
- 子活动编辑/删除后列表计数刷新；
- 去重保留后未保留推文及子活动退出业务列表；
- 周选择器、城市单选、预览和下载正确。

### 浏览器 E2E

- 真实导入一篇含多个活动的推文，只出现一行，详情显示全部活动和图片；
- 再次返回同一推文但更换 token，不产生新推文、活动、图片或失败日志；
- 批量审核后生成本周周报，Markdown/Excel 都按推文维度包含完整内容；
- 重复候选保留一方后，列表、详情、候选数和周报同步更新。

## 10. 验收标准

- 活动管理首页不再因一篇推文多活动而出现多行；
- 同一篇历史推文再次抓取能正常过滤，且过滤发生在重资源处理前；
- 精确重复不记失败，模糊重复可审核且没有同推文内误报或悬空候选；
- 详情完整展示原文、图片、OCR 和所有有效子活动；
- 周报只按本周推文、单城市导出，一篇推文在 Excel 中只占一行；
- 后端、前端组件和浏览器 E2E 全部通过；
- 同步更新数据库、API、UI、爬虫设计和测试案例文档。

## 11. 不在本次范围

- 不迁移 PostgreSQL、Redis 或 MinIO；
- 不支持跨城市合并周报；
- 不做自动无审核合并模糊重复推文；
- 不改变一期 Vue 3、FastAPI、Celery、SQLite 和本地文件存储技术栈。
