# 周报审核与按周导出设计

## 背景

活动管理会展示 `RAW`（待审核）和 `NEEDS_REVIEW`（待完善）记录，但周报只允许导出人工确认后的 `APPROVED` 活动。当前周报在没有已通过活动时仍生成空报告，且后端仅按城市和状态查询，没有真正限制 ISO 周次，容易让用户误解并可能混入其他周数据。

## 目标

- 保持抓取活动默认“待审核”，只有 `APPROVED` 活动进入周报。
- 周报严格按单个城市和所选 ISO 周次筛选。
- 所选城市和周次没有已通过活动时拒绝生成，并提示先审核活动。
- 活动管理支持选择多条活动后批量标记为“已通过”。

## 后端设计

`POST /api/v1/reports/generate` 将 `YYYY-Www` 解析为 ISO 周的周一零点，并使用下周一零点作为排他上界。查询条件同时包含城市、`status = APPROVED`、`start_time >= week_start` 和 `start_time < week_end`。周次格式非法时返回 422；结果为空时返回 422 和“所选城市和周次没有已通过活动，请先在活动管理中审核通过”。Markdown 与 Excel 使用同一城市、周次和审核状态条件，避免下载时重新混入其他日期数据。

新增 `POST /api/v1/activities/batch/approve`，请求体沿用批量操作的 `ids` 结构，限制 1 至 100 条。接口只查询未删除、未合并的活动，将选中记录更新为 `APPROVED`，返回 `approved_ids` 和 `approved_count`。已通过记录重复提交保持幂等。

## 前端设计

活动管理工具栏在“批量删除”之前增加 Element Plus 成功态按钮“批量通过”，使用 Element Plus 图标，不使用 emoji。按钮在未选中记录时禁用；点击后确认数量，调用批量通过接口，显示成功 Toast，清空选择并刷新当前分页。失败时显示后端错误。

周报页面继续由现有错误处理显示后端 422 文案。生成成功后只会出现至少包含一条已通过活动的报告。

## 测试与验收

- 后端测试覆盖 ISO 周边界、跨周排除、非已通过排除、空结果 422、非法周次 422、批量通过和幂等行为。
- 前端组件测试覆盖按钮禁用、确认、批量通过请求、成功 Toast、刷新和错误 Toast。
- 浏览器功能测试覆盖活动勾选并批量通过，再按相同城市和周次生成非空周报。
- 完整后端与前端测试、前端构建通过。
