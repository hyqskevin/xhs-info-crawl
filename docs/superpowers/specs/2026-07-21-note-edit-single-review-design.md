# 推文编辑与单条审核闭环设计

> 状态：已审核，依据 `AGENTS.md` 中的 TODO 持续执行授权自动进入开发。

## 1. 目标

活动管理支持编辑推文标题、正文、城市和发布时间，并在列表与详情中直接对单篇推文执行“通过”或“驳回”。原文链接保持只读，现有批量通过能力保持兼容。

## 2. 现状与根因

- `Note` 已保存标题、正文、城市、发布时间和审核状态，周报也会在生成时读取这些字段，因此不需要数据库迁移。
- 后端 `notes` API 只有列表、详情、批量通过和批量删除，没有推文更新及驳回入口。
- 前端活动管理的操作列只有“详情”，详情抽屉只允许编辑、删除子活动，无法形成推文级编辑和单条审核闭环。
- 审核状态已有 `PENDING`、`APPROVED`、`REJECTED` 语义，本次直接复用，不新增状态。

## 3. 方案选择

采用推文主资源上的两个明确接口：

- `PUT /api/v1/notes/{note_id}` 更新可编辑字段；
- `POST /api/v1/notes/{note_id}/review` 接收 `APPROVED` 或 `REJECTED`。

相比让前端用批量接口承载单篇操作，独立单篇接口能清楚表达意图、返回更新后的推文，并为驳回提供对称能力。相比把编辑与审核合并成一个通用 PATCH，分离接口可以避免普通内容编辑意外改变审核状态。

现有 `POST /api/v1/notes/batch/approve` 保留，继续服务批量通过。

## 4. 后端设计

### 4.1 更新推文

新增 `NoteUpdate` 请求模型：

- `title`: 必填非空，去除首尾空白，最大 512 字符；
- `content`: 必填，允许空字符串；
- `city_code`: 必填非空，必须对应已启用城市；
- `published_at`: 可空 ISO 8601 时间。

更新成功后提交事务并返回与详情一致的推文数据。`source_url` 不在请求模型中，因此客户端不能通过该接口修改原文链接。不存在、已删除或已合并的推文返回 404；城市无效返回 422。

### 4.2 单篇审核

新增 `NoteReviewRequest`，`status` 只允许 `APPROVED` 或 `REJECTED`。接口复用可见推文查询，修改 `review_status` 后提交并返回推文 ID 与最新审核状态。非法状态由请求校验返回 422，隐藏推文返回 404。

编辑内容不自动重置审核状态：运营人员可修正已通过或已驳回推文，再显式执行下一次审核，避免保存动作暗含状态变化。

## 5. 前端设计

### 5.1 列表操作

操作列提供“详情”“编辑”“通过”“驳回”。通过和驳回在确认框确认后调用单篇审核接口；成功显示 Toast 并刷新列表。当前已处于目标状态时不显示对应按钮，避免无意义重复操作。

### 5.2 详情操作

详情抽屉顶部展示审核状态，并提供“编辑推文”“通过”“驳回”。审核成功后同时刷新详情和列表，使两个视图保持一致。

### 5.3 编辑对话框

编辑表单包含标题、正文、城市、发布时间和只读原文链接。发布时间使用 Element Plus `datetime` 选择器，并以 ISO 8601 提交；清空时提交 `null`。标题和城市为必填，保存期间禁用重复提交。

保存成功后关闭对话框，刷新列表；若从详情打开，额外刷新详情。失败时由全局 HTTP 处理或局部错误 Toast 告知用户，对话框保持打开以便修正。

审核按钮的请求失败同样显示错误 Toast，且不提前改变本地状态。

## 6. 数据流与周报一致性

编辑直接修改 `Note` 聚合根。活动列表、详情和后续周报均在请求时从数据库读取 `Note`，因此更新后的标题、正文、城市和发布时间会自然进入这些消费者。已生成的历史周报内容保持快照语义，不在本次编辑时自动重写。

## 7. TDD 范围

### 后端

- 更新标题、正文、城市和发布时间后，列表、详情及数据库返回新值；
- `source_url` 不可通过更新请求修改；
- 无效城市、空标题、隐藏推文更新失败；
- 单篇通过和驳回分别写入正确状态；
- 非法审核状态失败，批量通过保持可用；
- 周报生成读取编辑后的推文内容。

### 前端组件

- 列表和详情均显示单篇编辑、通过、驳回入口；
- 编辑表单提交正确字段且不提交 `source_url`；
- 保存后刷新列表和详情并显示成功 Toast；
- 审核前显示确认框，成功刷新，失败显示错误 Toast；
- 已通过/已驳回状态隐藏对应同状态按钮；
- 批量通过能力不回退。

### 浏览器 E2E

- 从列表编辑推文并确认原文链接只读；
- 从详情通过单篇推文并看到状态刷新；
- 从列表驳回单篇推文并看到成功提示；
- 取消确认不发送审核请求。

## 8. 验收标准

- 编辑后列表、详情、数据库与后续新生成周报均使用更新后的推文信息；
- 原文链接不可编辑；
- 待审核推文无需批量勾选即可变更为已通过或已驳回；
- 列表和详情均可执行单篇审核；
- 审核操作有确认提示，编辑和审核均有明确成功/失败反馈；
- 原有批量通过继续工作；
- 后端、前端组件、前端构建和浏览器 E2E 通过。

- 不自动重写已经生成的历史周报；
- 不调整子活动审核状态或周报筛选规则，该事项由后续独立 TODO 处理；
- 不修改原文链接或抓取来源数据。
