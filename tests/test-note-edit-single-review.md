# 推文编辑与单条审核闭环测试案例

关联 spec：`docs/superpowers/specs/2026-07-21-note-edit-single-review-design.md`

## 后端 API

1. 建立一篇待审核推文和一个已启用城市，调用 `PUT /api/v1/notes/{id}` 修改标题、正文、城市和发布时间。
2. 验证响应、详情、列表和数据库均返回更新值；额外提交的 `source_url` 不改变原文链接。
3. 分别提交空白标题、停用城市和未知城市，验证返回 `422` 且原推文不变。
4. 分别调用单篇审核接口提交 `APPROVED`、`REJECTED`，验证数据库状态与响应一致。
5. 提交 `PENDING` 等不支持的目标状态，验证返回 `422`。
6. 回归批量通过接口，验证原能力不变且不修改子活动状态。
7. 编辑已通过推文后生成新周报，验证新标题、正文、城市和发布时间进入新报告；既有历史报告不自动重写。

## 前端组件

1. 活动管理列表每行显示“详情、编辑推文、通过、驳回”；已处于目标状态时隐藏同状态审核按钮。
2. 从列表打开编辑对话框，验证标题、正文、城市、发布时间可编辑，原文链接禁用。
3. 保存时只提交 `title/content/city_code/published_at`，成功后关闭对话框、刷新列表并显示 Toast。
4. 从详情打开编辑对话框，保存后同时刷新列表与详情。
5. 列表和详情的通过/驳回均先显示确认框；确认后刷新相应视图并显示成功 Toast。
6. 取消确认时不请求接口；API 失败时显示错误 Toast 且不乐观修改审核状态。
7. 批量通过仍需勾选后执行，行为不回退。

## 浏览器 E2E

1. 打开 `/activities`，从列表编辑推文标题；确认“原文链接”不可编辑，请求体不含 `source_url`，页面显示“推文已更新”。
2. 打开推文详情，点击“通过”并确认；验证提交 `APPROVED`，页面显示“推文已通过”。
3. 在列表点击“驳回”后取消，验证没有审核请求；再次点击并确认，验证提交 `REJECTED` 且显示“推文已驳回”。
